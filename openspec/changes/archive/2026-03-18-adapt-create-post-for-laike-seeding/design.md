## Context

当前 `plurk-mcp-create-post` 只描述了通用的新帖写入流程，重点在确认最终内容、再调用 `plurk_post`。这次要补的是内容策略层编排：让 skill 在 Laike 台湾区公开推广场景下，先读取本地角色素材，再按指定内容方向生成 Plurk 草稿，同时保留现有写入护栏。

这个改动主要落在技能说明与 agent 元数据，不涉及上游 `plurk-mcp` 工具面扩展。约束点有三类：

- 内容必须贴近台湾 Plurk 二次元语境，而不是通用繁中文案。
- 角色素材来自 workspace 本地的 Excel 文件，属于运行时上下文，不适合写死成仓库静态资料。
- 生成草稿和执行 `plurk_post` 必须继续分离，避免把“帮我写一条”滑成“直接帮我发”。
- 推广内容必须是公开披露的推荐或分享，不能伪装成普通用户自发发言，也不能建立在对未成年人或情绪脆弱人群的诱导上。

## Goals / Non-Goals

**Goals:**

- 让 `plurk-mcp-create-post` 明确支持 Laike 台湾区公开推广草稿生成。
- 要求 skill 在起草阶段先读取 workspace 根目录下的 `laike男性角色表.xlsx`，并按需要结合现有角色。
- 把四种内容方向、语言风格、字数、标签和透明披露规则写成稳定的技能约束。
- 保留并强化“最终稿确认后才允许调用 `plurk_post`”的两阶段流程。
- 让 `agents/openai.yaml` 的默认提示与新工作流一致。

**Non-Goals:**

- 不修改 `plurk_post` / `plurk_reply` 的工具契约或服务端护栏。
- 不把 skill 做成通用的 Excel 数据处理器，也不在规范中绑定具体列名。
- 不新增自动发布、批量生成、多帖排程或内容审核能力。
- 不实现伪装成普通用户碎碎念的隐性广告，也不支持以未成年人或情绪脆弱状态作为推广定向。

## Decisions

### 1. 只修改 `plurk-mcp-create-post` 的工作流要求，不新增独立 capability

这次变化本质上是现有 Create Post 行为的细化，不是新的产品面。主规范仍然挂在 `plurk-mcp-skills` 下，通过修改 Create Post requirement 来表达。

备选方案是新增一个独立的 `laike-seeding` capability，但这会把“内容策略”错误建模成一个新产品能力，反而弱化它和写入工作流的绑定关系，因此不采用。

### 2. 将 Excel 角色表视为运行时本地素材源，不在 spec 中硬编码表结构

skill 需要先读取 `laike男性角色表.xlsx`，但规范只要求“读取并提取可用男性角色素材”，不要求固定列名或工作表结构。这样可以兼容后续角色表增删字段，而无需频繁改 spec。

如果文件缺失或不可读，skill 应明确说明缺少本地素材，且不得编造“表内现有角色”；此时只能在不依赖表内角色的前提下继续起草，或等待用户补齐文件。

备选方案是在 spec 中规定具体列名，但当前仓库并不持有该文件，也没有稳定 schema，这样会把实现耦合到易变的外部文档，因此不采用。

### 3. 将“公开推广内容生成”拆成固定顺序：读素材、选主题、出草稿、确认发布、执行写入

新增内容策略后，最容易出现的偏差是 agent 直接从 brief 跳到写入。为此，skill 应显式要求以下顺序：

1. 读取角色表；
2. 随机选择主题，并匹配 A/B/C/D 内容方向；
3. 生成 20-200 字、台湾繁中、带 1-2 个标签且明确披露推广身份的 Plurk 草稿；
4. 若 agent 对文案有整理或补全，先展示最终稿并等待确认；
5. 只有在用户明确要求现在发布时，才调用 `plurk_post`。

备选方案是把“生成草稿”和“确认发布”合并成一个步骤，但这会削弱现有写入护栏，不采用。

### 4. 在 skill 本体中强制透明披露与非操纵性表达

四种内容方向、语言约束和推广披露方式只影响 `plurk-mcp-create-post`，不需要提升到共享护栏。实现上应把详细规则写入 `SKILL.md`，同时把 `agents/openai.yaml` 的 `short_description` 与 `default_prompt` 调整为“先生成/确认草稿，再写入”。

skill 还需要明确：

- 文案必须表明这是推荐、产品分享、体验分享或合作资讯之一。
- 不得伪装成普通用户自发碎碎念、匿名爆料或虚构个人经历。
- 如果写到陪伴、聊天灵感或娱乐向占卜玩法，只能作为功能介绍，不能写成医疗、心理治疗或保证准确的结果。

备选方案是把这类策略塞进 `skills/_shared/`，但共享层更适合工具契约和安全护栏，不适合承载单技能的营销内容策略。

## Risks / Trade-offs

- [本地 Excel 文件在不同环境中可能缺失] → 在 skill 中要求先检查并明确反馈缺失，不得假装读取成功或伪造表内角色。
- [台湾网感文案过于具体，后续可能需要频繁调整] → 将风格约束集中在 `SKILL.md`，规格层只固定可验证的行为边界与输出要求。
- [推广团队可能更偏好隐性原生广告] → 在 skill 中明确透明披露是硬约束，不因转化诉求而移除。
- [公开推广导向仍可能鼓励单回合直接发帖] → 保留“最终稿确认后才能调用 `plurk_post`”作为硬性流程要求。
- [随机选题会降低可预测性] → 仅要求“随机选择主题”，不要求完全随机实现；测试时验证主题来自允许方向即可。

## Migration Plan

1. 更新 `skills/plurk-mcp-create-post/SKILL.md`，加入 Laike 公开推广策略、角色表读取和草稿生成顺序。
2. 更新 `skills/plurk-mcp-create-post/agents/openai.yaml`，对齐新的默认提示。
3. 在 change spec 中修改 Create Post requirement，并校验 wording 与主规范一致。
4. 实现完成后运行 `openspec validate adapt-create-post-for-laike-seeding`。

本变更仅涉及文档与技能提示，不需要数据迁移。若需回滚，恢复相关 skill 文件和 spec 变更即可。

## Open Questions

- `laike男性角色表.xlsx` 的具体列结构尚未纳入仓库版本控制；实现阶段需要按实际文件内容确定读取粒度，但不应把这些细节固化进 spec。
