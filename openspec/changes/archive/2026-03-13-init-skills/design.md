## 背景

这个仓库当前只有 OpenSpec 脚手架。上游 `plurk-mcp` 项目已经为 OpenClaw 定义了一层边界明确的 MCP 执行层，包括 6 个 MCP 工具、受保护的写路径、本地审计日志，以及清晰的安全边界；其中 edit、delete、follow、fan-management 等操作被明确排除在外。

这个仓库当前要解决的核心问题不是 API 设计，而是操作层封装。OpenClaw 需要一组可复用的 skills，去指导 agent 以正确、一致、且符合服务端真实限制的方式使用现有 MCP 能力。如果技能包自行扩展、夸大或改写上游行为，就会很快与真实服务产生漂移，并带来安全风险。

## 目标与非目标

**目标：**
- 创建 `plurk-mcp` 的首版可复用技能包，使其可以被 OpenClaw 直接安装和触发。
- 让仓库结构本身适合公开发布，而不是依赖会话内私有目录。
- 首先覆盖一个统一入口 skill 和五个已确认的具体工作流：加载 Profile、加载 Alerts、加载 Mentions、创建 Post、创建 Reply。
- 保证技能说明与上游 `plurk-mcp` 的真实工具集合和护栏保持一致。
- 将工具说明、策略边界和工作流事实沉淀到共享参考资料中，减少技能之间的重复描述。
- 为每个技能提供元数据，使其可以在 OpenClaw/Codex 的技能界面中被发现。
- 以本地路径安装为首版默认接入方式，先打通最短使用路径。

**非目标：**
- 不在这个仓库里扩展 `plurk-mcp` 的 API 面，也不发明新的 MCP 工具。
- 不把内容策略、活动策划、发布时间安排等编辑层逻辑编码进 skills。
- 不支持 edit、delete、follow、fan-management 等当前未开放的高风险变更动作。
- 在首批安全工作流尚未验证前，不把仓库扩展为一个泛化的 Plurk 自动化技能库。
- 第一版不以远程 Git URL 安装兼容性为目标。

## 设计决策

### 使用一层共享参考资料，加上五个聚焦技能

首版技能包应当包含一层面向上游契约的共享参考资料，以及六个面向工作流的技能：
- `plurk-mcp-skills`
- `plurk-mcp-load-profile`
- `plurk-mcp-load-alerts`
- `plurk-mcp-load-mentions`
- `plurk-mcp-create-post`
- `plurk-mcp-create-reply`

这样做可以让统一入口 skill 负责路由，让具体 skill 保持紧凑、易触发，同时避免在多个技能中重复拷贝同一份工具清单和护栏说明。

备选方案：
- 一个大型 `plurk-mcp` 总技能：目录更简单，但触发精度差，而且会把很多分支逻辑塞进同一个文件。
- 每个 MCP 工具对应一个 skill：粒度更细，但过于底层，不贴近真实用户工作流，也更容易被错误触发。

### 以上游 plurk-mcp 仓库作为事实来源

共享参考资料和技能说明应当直接来自上游 `plurk-mcp` 的 README、tool catalog 和 OpenSpec 规格。本技能仓库不应该描述任何服务端实际并不存在的能力。

这意味着实现时必须明确记录：
- 当前支持的 6 个 MCP 工具
- 回帖资格边界
- 每日新帖配额行为，以及 reply 不占用每日新帖配额
- 同线程回复冷却行为
- 当前不支持的高风险动作

备选方案：
- 仅按产品设想写 skills：起步更快，但很容易与已实现的 MCP 服务发生漂移。
- 把大段上游文档直接复制进每个 skill：写起来简单，但上下文浪费严重，也更难维护。

### 写操作技能要具备上下文意识，而不是只做命令包装

`Create Post` 和 `Create Reply` 两个技能不应该只是 `plurk_post` 与 `plurk_reply` 的薄包装，而应该把安全工作流一起编码进去：
- 发帖技能在写入前先整理或确认要发布的内容
- 回帖技能在写入前先读取 thread context
- 两个技能都把策略拒绝视为正常边界，而不是需要绕过的错误

这样可以提升安全性，也能让 skill 真正服务于工作流，而不只是 API 调用层。

备选方案：
- 极短技能，只把用户请求映射成一次工具调用：虽然紧凑，但会遗漏关键的前后置流程。

### 每个技能都带上面向 OpenClaw 的元数据

每个要交付的技能都应包含必需的 `SKILL.md`，以及一个 `agents/openai.yaml` 文件。这样可以改善发现体验，并让首版技能包可以直接适配 OpenClaw/Codex 的安装和展示流程。

备选方案：
- 只提供 `SKILL.md`：虽然最小可用，但在 UI 发现和安装体验上会更弱。

### 使用仓库级 `skills/` 公共目录，而不是 `.codex/skills` 内部目录

首版实现应当把真正要发布的技能放在仓库级 `skills/` 目录下，并把共享参考放在 `skills/_shared/`。`.codex/skills` 更适合当前会话或开发期内部使用，不适合作为对外分发结构。

这样做有三个直接好处：
- 仓库本身就是一个可开源的 skill pack
- OpenClaw 可以通过本地路径直接加载该仓库
- 对外暴露的技能不会与内部开发辅助技能混在一起

由于 OpenClaw 已支持在扫描根目录时识别嵌套的 `skills/` 根目录，因此本仓库可以同时保留仓库级入口文件和公开子技能目录，而不牺牲本地路径加载能力。

备选方案：
- 继续把技能放在 `.codex/skills`：开发期方便，但不适合公开分发。
- 只保留根 `SKILL.md`，不拆分子技能：入口更简单，但会降低触发精度，也不利于后续扩展。

### 第一版以本地路径安装作为默认验收方式

第一版的验证目标应该尽量明确。由于这个仓库的直接目标是“在 OpenClaw 中可用”，首版应先以本地路径方式完成安装、发现和触发验证，而不是同时追求本地路径和远程 Git URL 两种接入方式。

这样可以让任务定义、目录布局和 smoke test 更稳定，也避免在还没有真实使用反馈之前为远程分发做过早设计。

备选方案：
- 同时要求本地路径和远程 Git URL 安装：覆盖更广，但会让首版验收标准变得分散。
- 暂不定义任何安装方式：实现阻力最小，但后续验证标准会变得模糊。

## 风险与权衡

- 共享参考资料可能与上游 `plurk-mcp` 发生漂移 -> 缓解方式：首版内容直接从上游 README、tool catalog 和 spec 提炼，且控制范围尽量小。
- 技能名称对真实 OpenClaw 提示词来说可能过窄或过宽 -> 缓解方式：首版先围绕具体工作流命名，后续再根据实际使用情况调整。
- 加载类技能与创建类技能可能存在部分语义重叠 -> 缓解方式：三个加载类技能保持显式只读，两个创建类技能只额外引入写操作相关规则。
- OpenClaw 的安装方式后续可能变化 -> 缓解方式：首版先采用公开 `skills/` 目录结构，并把 OpenClaw 元数据隔离在每个技能目录内。

## 落地计划

1. 创建共享参考文件，描述当前 `plurk-mcp` 的真实契约。
2. 新增五个首版技能目录，并补齐 `SKILL.md` 与 `agents/openai.yaml`。
3. 校验所有内部引用路径是否正确，以及技能内容是否仍然处于支持的 MCP 边界内。
4. 以仓库根目录作为本地路径，在 OpenClaw/Codex 中做一次安装和发现层面的 smoke test，再决定是否继续扩展更细化的技能。

## 待确认问题

- 首个版本是否还需要一个单独的 troubleshooting skill，用来处理凭证问题或策略拒绝诊断，还是先等真实使用痛点出现后再补？
